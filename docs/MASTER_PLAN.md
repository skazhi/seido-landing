## Часть 2. Честная оценка: реалистично ли с тобой и ИИ?

### Краткий ответ: **да, но не до конца**

Реалистично сделать:
- MVP (бот + база + календарь)
- Веб-сайт
- Первые 500–2000 пользователей
- Партнёрства с 3–10 организаторами
- Pro-подписку и комиссию со слотов
- Базовый маркетплейс

Сложно или нереалистично в одиночку:
- Полноценное мобильное приложение (особенно iOS)
- Сложные интеграции (Garmin, Strava, банки)
- Крупные рекламные контракты (нужны продажи, переговоры)
- Масштабирование на 10 000+ активных без команды
- Юридическая экспертиза спорных кейсов

---

| Задача | Человек | ИИ |
|--------|---------|-----|
| Архитектура продукта, приоритеты | ✅ | Помощь в структурировании |

| Дизайн | Финальные решения | ✅ Черновики, идеи |
| Тексты (посты, оферта) | Финал, тон | ✅ Черновики |

| Юридические документы | ✅ Проверка юристом | Черновики |
| Маркетинг, SMM | ✅ | Контент-план, тексты |
| Поддержка пользователей | ✅ | FAQ, шаблоны ответов |
| Монетизация, стратегия | ✅ | Анализ, варианты |
| Тестирование в реальных условиях | ✅ | Автотесты, скрипты |

ИИ заметно ускоряет разработку и помогает с рутиной. Критичные решения — за тобой.

---

### Вывод по реалистичности

- **Фазы 0–3:** реалистично за 4–6 месяцев при 10–15 часах в неделю.
- **Фаза 4 (маркетплейс):** реалистично за ещё 3–4 месяца, но нужны живые переговоры с организаторами.
- **Фазы 5–6:** частично реальны; полный масштаб потребует команды или инвестиций.

Идея сильная, MVP и ранний рост — достижимы в связке «ты + ИИ».

## Часть 3. Трудности

### Технические

| Проблема | Решение |
|----------|---------|
| Парсинг протоколов (разные форматы) | Универсальные парсеры + ручная проверка сложных кейсов |
| Масштаб БД (много результатов) | Индексы, пагинация, кеш (Redis при росте) |
| Падение бота/сайта | Мониторинг, автоматический рестарт, резервный хостинг |
| Миграции БД при изменениях схемы | Версионированные миграции (Alembic и т.п.) |

### Продуктовые

| Проблема | Решение |
|----------|---------|
| Курица-яйцо: мало забегов → мало пользователей | Начать с ручного наполнения, фокус на своём городе/регионе |
| Организаторы не идут в маркетплейс | Показывать трафик, конверсию; начинать с малой комиссии |
| Пользователи не возвращаются | Уведомления, полезный контент, личный календарь |
| Перегруженность фичами | Жёсткий приоритет: календарь, результаты, слоты. Всё остальное — позже |

### Операционные

| Проблема | Решение |
|----------|---------|
| Всё держится на одном человеке | Документировать процессы; иметь план «если заболел/выгорел» |
| Много мелких задач | Чеклисты, недельные цели, «не больше 3 приоритетов» |
| Выгорание | Регулярные паузы, реалистичные дедлайны |
| Отвлечение на «хотелки» | Фиксированный бэклог, отклонять или откладывать |

### Маркетинговые

| Проблема | Решение |
|----------|---------|
| Мало бюджета на рекламу | Органика: комьюнити, сарафан, контент в канале |
| Сложно выделиться | Честность, фокус на пользе, «всё в одном месте» |
| Конкуренция с крупными | Ниша: Россия, бег, экосистема. Скорость и гибкость как преимущество |

## Часть 4. Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Проект заброшен (выгорание, жизнь) | Средняя | Высокое | Чёткий MVP, короткие циклы, «минимум для запуска» |
| Организаторы отказывают в партнёрстве | Средняя | Среднее | Парсинг публичных данных как fallback; показывать ценность (трафик) |
| Мало пользователей | Средняя | Высокое | Фокус на 200+ знакомых, сарафан, полезный контент |
| Юр.претензии (ПД, авторские права) | Низкая | Высокое | Следование 152-ФЗ, оферта, удаление по запросу |
| Техдолг, сложность поддержки | Высокая | Среднее | Простой стек, меньше «умных» решений |
| Конкуренция (крупный игрок) | Средняя | Среднее | Быстрое развитие, сильное сообщество |
| Нехватка денег на инфраструктуру | Низкая | Среднее | Бесплатные тарифы (Vercel, SQLite/Postgres на дешёвом VPS) |

## Часть 7. Итоговый чеклист «старт»

Перед тем как углубляться:

- [ ] Есть 10–15 часов в неделю минимум на 3–4 месяца
- [ ] Готов к тому, что первые деньги могут появиться через 6–12 месяцев
- [ ] Доступ к 30–50 бегунам для теста
- [ ] Готов сам вести переговоры с организаторами
- [ ] Домен и хостинг не создадут финансовой нагрузки
- [ ] Есть запас на «провал» (время и небольшие деньги) без катастрофы

Если большинство пунктов «да» — можно начинать. Если нет — имеет смысл упростить план или сдвинуть сроки.

## Часть 10. До какого этапа — «Владимир + ИИ»?

### Можно справиться полностью (без команды)

- **Фаза 0:** Фундамент
- **Фаза 1:** MVP бот + база
- **Фаза 2:** Расширение базы и сообщества
- **Фаза 3:** Веб и календарь (включая Pro-подписку)

### Частично (лучше привлечь помощь)

- **Фаза 4:** Маркетплейс — переговоры с организаторами и юр.договоры удобнее вести с поддержкой (юрист, продажи). Технически — по-прежнему возможно соло.
- **Фаза 5:** Расширение — мобильное приложение (особенно iOS), интеграции — объём работ большой; можно часть отдать на аутсорс.

### Практический порог «соло + ИИ»

**Рекомендуемый момент для привлечения команды:** после выхода маркетплейса (фаза 4), когда есть первые 50–100 продаж слотов в месяц и 500+ активных пользователей.

До этого этапа — связка «Владимир + ИИ» достаточна. Команду имеет смысл добавлять, когда операционка (поддержка, контент, продажи) начинает забирать больше времени, чем разработка.

## Часть 11. Парсеры и перенос на сайт / другие каналы

### Принцип: данные — общие, клиенты — разные

Парсеры и сбор анонсов **не привязаны** к боту или сайту. Они пишут в единое хранилище (БД).

```
[Парсеры результатов] ──┐
[Парсеры анонсов]     ──┼──▶ [БД / API] ──▶ [Telegram-бот]
[Ручной ввод]         ──┘         │
                                  ├──▶ [Сайт]
                                  ├──▶ [VK-бот]
                                  ├──▶ [VK-канал / посты]
                                  └──▶ [Email-рассылка]
```

### Что делать уже сейчас

1. **Парсеры** возвращают нормализованные данные (как в `base.py`) и пишут в БД.
2. **Бот** читает из БД — не из парсера напрямую.
3. **API** (FastAPI) — единая точка доступа к данным. Сайт, VK, email — только клиенты API.

### Перенос «на сайт» и в другие соцсети

- **Переписывать парсеры не нужно.** Они уже обновляют общую БД.
- **Сайт** — новый клиент: делает запросы к API и отображает данные.
- **VK, Telegram-канал, рассылка** — тоже клиенты API: берут забеги/результаты и форматируют под свой канал.

Единая архитектура: **парсеры → БД → API → клиенты (бот, сайт, VK, email)**. Новые каналы = новые клиенты API, без переписывания парсеров.

## Часть 12. Архитектура «с нуля»: роли и структура

### Зачем продумывать структуру заранее

Чтобы не переделывать позже: разделение по ролям, единые модели данных и API с самого начала.

### Роли в системе

| Роль | Доступ | Где реализуется |
|------|--------|------------------|
| **Бегун** | Профиль, результаты, календарь, покупка слотов, подписки | Бот, сайт, приложение |
| **Организатор** | Кабинет: забеги, протоколы, продажи, аналитика | Сайт (отдельный раздел / поддомен) |
| **Тренер / специалист** | Каталог, профиль, отзывы | Сайт (позже) |
| **Админ** | Модерация, удаление, настройки | Бот (админ-команды), админка на сайте |
| **Гость** | Просмотр календаря, лендинг | Сайт, без регистрации |

### Структура продукта (блоки)

```
seido/
├── core/                    # Ядро (общее для всех)
│   ├── models/              # Модели данных (runner, race, result, subscription)
│   ├── api/                 # REST API (FastAPI)
│   └── db/                  # Схема БД, миграции
│
├── parsers/                 # Парсеры (результаты, анонсы)
│   ├── results/             # Парсеры протоколов
│   └── announcements/       # Парсеры анонсов
│
├── clients/                 # Клиенты (потребляют API)
│   ├── telegram_bot/        # Telegram-бот
│   ├── web/                 # Сайт (React/Vue)
│   ├── vk_bot/              # VK-бот (позже)
│   └── notifications/       # Рассылки (Telegram, email)
│
├── roles/                   # Логика по ролям
│   ├── runner/              # Сценарии бегуна
│   ├── organizer/           # Сценарии организатора
│   └── admin/               # Админ-сценарии
```

### Каналы коммуникации (единый контент, разные форматы)

| Канал | Формат | Источник контента |
|-------|--------|-------------------|
| Telegram-канал | Посты, анонсы | API (забеги) + шаблоны |
| VK | Посты, анонсы | Те же данные, другой шаблон |
| Email | Рассылка | Те же данные, HTML-шаблон |
| Push (приложение) | Уведомления | Те же данные |

Один источник данных (API), разные шаблоны под канал — без дублирования логики.

### Чеклист «архитектура на старте»

- [ ] БД: единая схема для runners, races, results, subscriptions
- [ ] API: эндпоинты не завязаны на бота (универсальные)
- [ ] Парсеры: пишут в БД, не в бота
- [ ] Бот: читает через API или напрямую из БД (на старте допустимо)
- [ ] Роли: в модели пользователя поле `role` (runner, organizer, admin)
- [ ] Масштаб: поддомен или путь `/organizer` для кабинета организатора
